1 — The All-Pairs Shortest Paths Problem

Given a weighted directed graph G = (V, E) with weight function w: E → ℝ, the all-pairs shortest paths (APSP) problem asks us to find the shortest path (minimum total weight) between every pair of vertices (i, j).

What we want

A V×V matrix D where D[i][j] = the weight of a shortest path from vertex i to vertex j, for all i and j. Plus a predecessor matrix Π that lets us reconstruct the actual path.

Applied scenario (logistics routing): A company manages routes between warehouses (vertices) and shipping lanes (directed edges). Each lane has a cost; rebates can make some edges negative. Input: a weighted directed graph and the vertex set. Output: D and Pi so every pair (i,j) has its minimum total cost and a reconstructable path. Constraints: no negative cycles (otherwise costs are unbounded below). Objective: precompute all-pairs shortest paths so any routing query is answered instantly.

Real-world applications: routing protocols (OSPF, BGP), GPS navigation with multiple destinations, network reliability analysis, arbitrage detection in currency exchange.

Try it yourself: Click "CLRS Example" in the controls, then click Build Steps and Step through the algorithm. Watch how the distance matrix evolves as each vertex k is used as an intermediate.

2 — Naive Approach: Run Dijkstra V Times

One idea: for each vertex s, run Dijkstra's algorithm from s to find the shortest paths from s to all other vertices. This solves APSP in O(V · (V+E) log V) time.

Problem: Dijkstra requires non-negative edge weights. If the graph has negative weights (but no negative cycles), we can use Bellman-Ford instead: O(V · VE) = O(V²E). For dense graphs where E = Θ(V²), this is O(V⁴) — very slow!

Floyd-Warshall's advantage

Floyd-Warshall handles negative weights, works on dense graphs in O(V³), and has a simple, elegant implementation. For dense graphs, it beats running Dijkstra V times.

When not to use Floyd-Warshall: If you only need shortest paths from one source, use Dijkstra (non-negative weights) or Bellman-Ford (negative weights). If the graph is sparse and you need all pairs, Johnson's is usually faster. Use Floyd-Warshall when you need all pairs and the graph is dense or has negative edges.

3 — The Dynamic Programming Idea

Floyd-Warshall uses dynamic programming. The key insight is to think about the intermediate vertices on a shortest path.

Consider the shortest path from vertex i to vertex j. Either this path:

  • Goes directly from i to j without passing through vertex k, or
  • Goes through vertex k: i → ... → k → ... → j (where the sub-paths i→k and k→j only use intermediate vertices from {1, ..., k-1})

Optimal Substructure: If a shortest path from i to j passes through vertex k, then the i→k portion and the k→j portion must themselves be shortest paths (using only vertices {1,...,k-1} as intermediates). This is the classic optimal substructure property that makes DP applicable.

4 — The Recurrence

Let d(k)[i][j] = the weight of a shortest path from i to j using only vertices {1, 2, ..., k} as intermediate vertices.

Base case (k=0): No intermediate vertices allowed. d(0)[i][j] = w[i][j] (direct edge weight, or ∞ if no edge).

The Floyd-Warshall Recurrence

d(k)[i][j] = min( d(k-1)[i][j],   d(k-1)[i][k] + d(k-1)[k][j] )

Either we don't use k as intermediate (first term), or we go through k (second term). We take the minimum.

The final answer is d(V)[i][j] — the shortest path using any vertex as intermediate. In practice, Floyd-Warshall updates the matrix in-place (it's provably equivalent).

Correctness invariant

After finishing the outer loop for k, D[i][j] equals the shortest path from i to j that uses only intermediate vertices in {1..k}. Base case k=0: D is just W (direct edges). Inductive step: when we consider vertex k, we either keep the best path without k or improve via k, so the invariant holds for k+1. If no path exists, D[i][j] stays INF (infinity) and Pi[i][j] stays NIL.

5 — The Algorithm

FLOYD-WARSHALL(W, n): // Initialize distance matrix D ← copy of W -- d[i][j] = w[i][j] initially π[i][j] ← i if w[i][j] < ∞ -- predecessor: came from i π[i][j] ← NIL otherwise // Triple nested loop — the heart of the algorithm for k = 1 to n: -- try each vertex as intermediate for i = 1 to n: for j = 1 to n: if d[i][k] + d[k][j] < d[i][j]: -- going via k is shorter? d[i][j] ← d[i][k] + d[k][j] -- update with shorter distance π[i][j] ← π[k][j] -- path now goes via k return D, Π

The algorithm is deceptively simple: just three nested loops, one comparison, and one update. The magic is in the ordering — by iterating k from 1 to n, we build up from shorter paths to longer ones.

6 — Predecessor Matrix & Path Reconstruction

The predecessor matrix Π[i][j] stores the predecessor of j on the shortest known path from i to j. Initially, if edge (i,j) exists, then Π[i][j] = i (j's predecessor on the direct path is i).

During the algorithm, if we find that going via k improves d[i][j], we set Π[i][j] = Π[k][j]. This records that the predecessor of j on the i→j path is the same as the predecessor of j on the k→j path.

Reconstructing a Path

To find path from i to j: follow Π[i][j] backward until you reach i. More precisely: start at j, then look at Π[i][j] to get previous node, then Π[i][prev], and so on until you reach i.

Try it: After running the algorithm, use the "Show Path" controls to visualize any shortest path in the graph.

7 — Negative Cycles

A negative cycle is a cycle whose total edge weight is negative. If a negative cycle exists on any path from i to j, then the shortest path from i to j is −∞ (we can loop around the cycle indefinitely).

Detection: After Floyd-Warshall completes, check the diagonal: if d[i][i] < 0 for any vertex i, then vertex i is on a negative cycle. The shortest-path problem has no solution (paths are −∞).

Try the "Negative Cycle" preset to see this in action — vertices on the cycle will show negative diagonal values highlighted in red.

📚 Lecture Notes — Modular Exponentiation

1. Problem Setup 2. Binary Exponent Idea 3. L→R and R→L 4. Correctness Intuition 5. RSA Example 6. Complexity 7. Common Pitfalls

1 — Problem: Compute a^b mod n Efficiently

Directly computing a^b is expensive and can overflow quickly. We only need the result modulo n, so we reduce after each operation.

Key Principle

(x * y) mod n = ((x mod n) * (y mod n)) mod n. This allows us to keep numbers small throughout the algorithm.

2 — Binary Exponentiation

Write exponent b in binary. Each bit determines whether we multiply the current result by the current power. This transforms exponentiation into a sequence of squaring operations and occasional multiplies.

Bit Interpretation

If b = b_t...b_1b_0 in binary, then a^b = product over i where b_i=1 of a^(2^i). Squaring updates powers of two efficiently.

Complexity: Naive repeated multiplication takes O(b), while binary exponentiation takes O(log b).

3 — Two Variants

VariantDirectionMain Loop Idea
Left-to-RightMSB → LSBSquare each step, multiply when bit is 1
Right-to-LeftLSB → MSBMultiply when bit is 1, square base each step

Both are correct and efficient. This tool lets you switch methods and compare operation flow.

4 — Correctness Intuition

At each step, an invariant is preserved: the processed bits have already contributed correctly to the current result modulo n.

Invariant (L→R)

After processing prefix bits b_t...b_i, result equals a^(value of that prefix) mod n.

Invariant (R→L)

result accumulates contributions from processed low-order bits, while base tracks a^(2^k) mod n at iteration k.

Example: 7^13 mod 23 13 (binary) = 1101 L→R sketch: result=1 bit 1: square -> 1, multiply -> 7 bit 1: square -> 3, multiply -> 21 bit 0: square -> 4 bit 1: square -> 16, multiply -> 20 Final value = 20

5 — RSA-Style Example

RSA encryption/decryption repeatedly computes m^e mod n or c^d mod n. Efficient modular exponentiation is the core operation.

Try preset: base=65, exponent=17, modulus=3233 (common teaching RSA parameters).

6 — Complexity and Operation Counts

MethodTimeTypical OpsSpace
Naive multiplicationO(b)b-1 multipliesO(1)
Binary exponentiationO(log b)~log2(b) squarings + conditional multipliesO(1)

This tool displays live counters for squarings and multiplies so students can observe the logarithmic growth directly.

7 — Common Pitfalls

Pitfall 1: Forgetting to apply modulo after each multiply/square causes overflow and wrong results.

Pitfall 2: Mixing bit direction (MSB→LSB vs LSB→MSB) without adjusting update logic.

Pitfall 3: Using modulus n ≤ 1. Modular arithmetic requires n > 1 for meaningful residue classes.

📚 Lecture Notes — FFT / DFT

1. Why Frequency Domain? 2. DFT Definition 3. Cooley-Tukey Idea 4. Butterfly Math 5. Inverse DFT 6. Complexity + Worked Example

1 — Why Go to the Frequency Domain?

Many signals look complicated in time-domain samples x[n], but become simple in frequency-domain bins X[k]. A pure sine wave often appears as a clear peak at one frequency bin, while mixtures show multiple peaks.

Interpretation

Time domain answers: "What is the signal value now?" Frequency domain answers: "Which oscillations are present, and how strong are they?"

Problem statement: Input: sampled signal x[0..N-1] (real or complex). Output: frequency coefficients X[0..N-1] and magnitudes |X[k]|. Objective: reveal dominant frequencies and enable efficient filtering or analysis. Constraints: for this tool, N is a power of 2 (radix-2 FFT), up to 128 for interactive playback.

2 — DFT Definition

The Discrete Fourier Transform maps N samples x[0..N-1] to N frequency coefficients X[0..N-1].

Forward Transform

X[k] = sum(n=0..N-1) x[n] * e^(-2*pi*i*k*n/N)

Each X[k] is a correlation with a complex sinusoid. Large |X[k]| means that frequency is strongly present.

Polynomial view

Define P(z) = sum(n=0..N-1) x[n] z^n. Then X[k] = P(omega_N^k), where omega_N = e^(-2*pi*i/N). The inverse DFT performs interpolation to recover x[n].

Naive DFT cost: N outputs × N terms each = O(N²). For large N this quickly becomes expensive.

3 — Cooley-Tukey FFT: The Core Insight

FFT reuses repeated terms by splitting the sequence into even and odd indices recursively (or iteratively with bit-reversal + staged butterflies).

Divide and Combine

Compute smaller transforms, then combine using twiddle factors W_N^k = e^(-2*pi*i*k/N). This reduces work from O(N²) to O(N log N).

In this tool, we use iterative Cooley-Tukey: reorder by bit-reversal, then process stages with butterfly width m = 2, 4, 8, ... N.

4 — Butterfly Operation (Single Update Unit)

Every stage applies many butterflies. For a pair (u, v) and twiddle w:

t = w * v top_new = u + t bot_new = u - t where u = x[k + j] v = x[k + j + m/2] w = exp(-2*pi*i*j/m)

This one pattern is repeated across all blocks and stages. The visualization panel "Current Butterfly" shows these exact values at each step.

5 — Inverse DFT (Reconstruction)

IFFT reconstructs the original signal from X[k]. If implementation is correct, IFFT(FFT(x)) returns x (up to tiny floating-point error).

Inverse Formula: x[n] = (1/N) * sum(k=0..N-1) X[k] * e^(+2*pi*i*k*n/N)

The "Inverse DFT Check" panel reports reconstruction error so you can verify numerical correctness during demos.

6 — Complexity and Worked Signal Example

MethodTimeMain IdeaWhen Useful
Naive DFTO(N²)Direct sum for each kSmall N, reference checks
FFT (Cooley-Tukey)O(N log N)Reuse via butterfliesReal-time and large-N transforms
Why O(N log N)

There are log2(N) stages, and each stage processes N/2 butterflies. Each butterfly does O(1) work (1 complex multiply + 2 add/sub), so total work is O(N log N).

Recurrence view: T(N) = 2T(N/2) + O(N) from splitting into even/odd parts and combining with twiddles, which solves to O(N log N).

Worked intuition: for N = 16 and x[n] = sin(2*pi*2*n/16), the strongest magnitude bins should appear near k = 2 and its symmetric partner k = 14.

When FFT is worth it: Use FFT for large N or repeated transforms. For very small N or when you only need a few bins, a direct DFT can be simpler.

Suggested demo flow: Generate sine (N=16, freq=2), Build Steps, Play. Then switch to square and chirp to compare broader spectra.
